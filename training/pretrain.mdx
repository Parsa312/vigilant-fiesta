## Overview

Pretraining on Lightning AI runs on auto-provisioned GPU clusters. You write a standard PyTorch Lightning `Trainer` script; the platform handles node provisioning, inter-node networking (NCCL over InfiniBand), checkpointing to shared storage, and teardown.

A typical pretraining workflow:

1. Prepare and upload your dataset to a Teamspace storage bucket.
2. Write or adapt a training script using `lightning.Trainer`.
3. Submit the job with a cluster configuration (number of nodes, GPU type).
4. Monitor loss curves and system metrics in the Jobs dashboard.
5. Retrieve checkpoints from storage for evaluation or deployment.

## Cluster Configuration

Jobs are configured via a YAML spec or CLI flags. A minimal multi-node config:

```yaml
# pretrain_job.yaml
name: llm-pretrain-run
nodes: 8
accelerator: A100-80GB
devices_per_node: 8
image: lightning/pytorch:2.3-cuda12
entrypoint: "python train.py --config configs/llm_7b.yaml"
storage:
  - mount: /data
    bucket: s3://my-org-training-data
  - mount: /checkpoints
    bucket: s3://my-org-checkpoints
```

Submit with:

```bash
lightning jobs submit pretrain_job.yaml
```

## Checkpointing and Fault Tolerance

Lightning AI automatically enables asynchronous checkpointing to shared storage. If a node fails mid-run:

- The job pauses and waits up to 10 minutes for the node to recover.
- If recovery fails, the job restarts from the last saved checkpoint.
- Checkpoint frequency defaults to every 1,000 steps; override with `--checkpoint_every N`.

To resume a cancelled or failed job from the latest checkpoint:

```bash
lightning jobs resume <job-id>
```
